Tópico 1: Mapeamento MER $\to$ Modelo Relacional

O objetivo é traduzir o Modelo Entidade-Relacionamento (MER) para o Modelo Relacional (MR).

Regras Essenciais de Mapeamento:

  • Entidade Forte: Vira Tabela. A PK da entidade vira PK da tabela.
  • Entidade Fraca: Vira Tabela. A PK é composta por: (FK da entidade proprietária + Chave Parcial da entidade fraca).
  • Relacionamento 1:1: Chave Estrangeira (FK). A PK de uma entidade migra para a tabela da outra (preferencialmente para o lado com participação total).
  • Relacionamento 1:N: Chave Estrangeira (FK). A PK do lado '1' sempre migra como FK para a tabela do lado 'N'.
  • Relacionamento M:N: Nova Tabela Associativa. A PK desta nova tabela é composta pelas FKs de ambas as entidades participantes.
  • Atributo Multivalorado: Nova Tabela. A PK é composta por: (FK da entidade original + o próprio atributo).
  • Relacionamento N-ário ($n > 2$): Nova Tabela Associativa. Similar ao M:N, incluindo as PKs de todas as $n$ entidades como FKs.

Tópico 2: Normalização

Objetivo: Reduzir redundância e eliminar anomalias.

Anomalias (Problemas da Não Normalização):

  1. Inserção: Não é possível cadastrar uma nova agência (lado '1') até que ela tenha um funcionário (lado 'N').
  2. Atualização: Mudar o endereço de uma agência exige atualizar o registro de todos os seus funcionários. Se um falhar, gera inconsistência.
  3. Exclusão: Ao demitir o último funcionário de uma agência, os dados da própria agência (que estavam na mesma linha) são perdidos.

Dependência Funcional (DF):

  • Definição: $A \to B$ (A determina B). Para cada valor de A, existe apenas um valor de B.
  • Parcial: Atributo não-chave depende de apenas parte de uma PK composta. (Resolvido na 2FN).
  • Transitiva: Atributo não-chave depende de outro atributo não-chave. ($A \to B \to C$). (Resolvido na 3FN).

Formas Normais (FN):

  • 1FN: Atributos devem ser atômicos (indivisíveis). Não pode ter atributos multivalorados.
  • 2FN: Estar na 1FN + Não ter dependências parciais. (Relevante apenas para tabelas com PK composta).
  • 3FN: Estar na 2FN + Não ter dependências transitivas.
  • BCNF (Forma Normal de Boyce-Codd): Mais rigorosa que a 3FN. Para toda DF $X \to Y$, $X$ deve ser uma superchave.

Tópico 3: Linguagens de Consulta (SQL)

DDL (Data Definition Language) – Estrutura

  • CREATE TABLE: Define tabelas e Constraints (Restrições).
  • Ações de Integridade Referencial (FK):
    • ON DELETE RESTRICT / NO ACTION (Padrão): Bloqueia a exclusão do "pai" se ele tiver "filhos".
    • ON DELETE CASCADE: Propaga a exclusão. Excluir o "pai" exclui automaticamente os "filhos".
    • ON DELETE SET NULL: Excluir o "pai" atualiza a FK dos "filhos" para NULL.

DML (Data Manipulation Language) – Dados

  • INSERT: Adiciona novas linhas.
  • UPDATE: Modifica linhas existentes.
  • DELETE: Remove linhas. (Cuidado: DELETE FROM tabela sem WHERE apaga tudo).

DQL (Data Query Language) – Consulta (SELECT)

  • Junções (JOINs):
    • INNER JOIN: Apenas linhas com correspondência em ambas as tabelas.
    • LEFT JOIN: Todas as linhas da tabela da Esquerda + correspondências da Direita (NULL onde não houver).
    • RIGHT JOIN: Todas as linhas da tabela da Direita + correspondências da Esquerda.
  • Agregação:
    • GROUP BY: Agrupa linhas para usar funções (COUNT, SUM, AVG, MAX, MIN).
    • HAVING: Filtra grupos (é o WHERE do GROUP BY).

Armadilha de Concurso: LEFT JOIN + WHERE

Cenário: Listar todos funcionários (A) e contar apenas suas chamadas "outgoing" (B).

Consulta ERRADA:


SELECT A.name, COUNT(B.call_id)
FROM EmployeeTable A
LEFT JOIN CallTable B ON A.name = B.call_from_name
WHERE B.call_type = 'outgoing'  -- <--- ARMADILHA
GROUP BY A.name;

Por quê? O WHERE é aplicado depois do LEFT JOIN. Ele filtra as linhas onde $B.call\_type$ é NULL (funcionários sem chamadas), transformando o LEFT JOIN em um INNER JOIN e eliminando funcionários do resultado.

Consulta CORRETA (Mover o filtro para o ON):


SELECT A.name, COUNT(B.call_id)
FROM EmployeeTable A
LEFT JOIN CallTable B ON A.name = B.call_from_name
                   AND B.call_type = 'outgoing' -- <--- CORRETO (Filtra na junção)
GROUP BY A.name;

Tópico 4: SGBDR e Propriedades ACID

  • SGBDR (Sistema de Gerenciamento de BD Relacional): O software (Ex: MySQL, PostgreSQL, Oracle) que implementa o modelo relacional.
  • Propriedades ACID: Garantem a confiabilidade das transações.
    1. A - Atomicidade (Atomicity): "Tudo ou nada". A transação ou é completada (COMMIT) ou desfeita (ROLLBACK).
    2. C - Consistência (Consistency): A transação leva o banco de um estado válido para outro estado válido, respeitando todas as regras.
    3. I - Isolamento (Isolation): Transações concorrentes não interferem umas nas outras. O resultado é como se fossem executadas em sequência.
    4. D - Durabilidade (Durability): Uma vez que a transação é confirmada (COMMIT), ela é permanente, mesmo em caso de falha.

Tópico 5: Segurança em Banco de Dados

Controle de Acesso (DCL - Data Control Language):

  • GRANT: Concede privilégios (Ex: GRANT SELECT ON clientes TO josemar;).
  • REVOKE: Remove privilégios (Ex: REVOKE SELECT ON clientes FROM josemar;).
  • WITH GRANT OPTION: Permite que o usuário que recebeu a permissão também a conceda a outros.
  • REVOKE... CASCADE: Revoga a permissão do usuário E de todos os outros para quem ele a concedeu.

Criptografia:

  • Em Repouso (At-Rest): Protege os arquivos de dados físicos no disco (Ex: TDE).
  • Em Trânsito (In-Transit): Protege a comunicação na rede (Ex: SSL/TLS).

Auditoria:

  • Conceito: Monitorar e registrar ações (quem fez o quê, quando) para prevenir ou detectar violações.
  • Auditoria C2: Modo de auditoria (obsoleto, mas cobrado) que registra "tudo ou nada", como logins (sucesso/falha), criação/exclusão de objetos e chamadas de procedures.

Tópico 6: Bancos de Dados NoSQL ("Not Only SQL")

Alternativa aos SGBDRs para lidar com Big Data (Volume, Variedade, Velocidade).

Trade-off Central: ACID vs. BASE

  • Relacional (SGBDR) = ACID (Foco na Consistência).
  • NoSQL = BASE (Foco na Disponibilidade).
    • Basically Available (Basicamente Disponível).
    • Soft State (Estado "Suave" - pode mudar sem intervenção).
    • Eventual Consistency (Consistência Eventual). O sistema eventualmente fica consistente; leituras imediatas após escritas podem retornar dados antigos.

4 Tipos Principais de NoSQL:

  1. Orientados a Documentos: Armazena em JSON/BSON. (Ex: MongoDB).
  2. Chave-Valor: Dicionário de alta performance. (Ex: Redis).
  3. Orientados a Colunas (Column-Family): Otimizado para Big Data e IoT. (Ex: Cassandra).
  4. Orientados a Grafos: Foco nos relacionamentos (Nós e Arestas). (Ex: Neo4j).
